## What this is
A self-hosted **HRIS software** (Laravel 10 / PHP 8.1, Blade + Bootstrap 5, jQuery/axios) sold to PH companies to run in-house. Covers payroll, attendance, scheduling, leave & OT, government compliance (SSS/PhilHealth/Pag-IBIG/BIR), 201 files, loans, role-based access, audit trail, reports — plus **KuBo**, an internal employee community/social feed.

## Commands
- **Install:** `composer install` then `npm install`
- **Front-end build:** `npm run dev` (Laravel Mix, alias of `npm run development`) · `npm run watch` (rebuild on change) · `npm run prod` (minified production build)
- **Run app locally:** `php artisan serve`
- **Tests (PHPUnit):** `php artisan test` or `./vendor/bin/phpunit`
  - Single suite: `php artisan test --testsuite=Unit` (suites are `Unit` and `Feature`)
  - Single test: `php artisan test --filter NightDiffTest`
- **Migrations:** `php artisan migrate`
- **Permissions:** `php artisan app:create-permission` — **required** to register new permission keys (see Access control). Run after adding a key, then assign it to roles in the UI.
- **Clear caches:** `php artisan optimize:clear` — then **restart PHP / the web server** for changes to take effect (per `ACTIVATION_CHECKLIST.md`).

### Activation flow (from `ACTIVATION_CHECKLIST.md`)
After deploying changes: `php artisan migrate` → `php artisan app:create-permission` → `php artisan optimize:clear` + restart PHP → assign permissions in the Roles screen → re-compute payroll once.

## Tech stack & key packages
- **Laravel 10**, PHP `^8.1`, MySQL (`DB_CONNECTION=mysql`, default db `projinventory`). Queue/cache/session default to sync/file; broadcast = log.
- **`spatie/laravel-permission`** — role/permission backbone (see Access control below).
- **`maatwebsite/excel`** — the bulk import/export engine (Exports in `app/Exports`, import services in `app/Services`, templates `*_import_template.xlsx` at repo root).
- **`doctrine/dbal`** (schema changes), **`laravel/sanctum`** (API auth), **`laravel/tinker`**.
- **Front end:** Laravel Mix (not Vite). Blade + Bootstrap 5 + FontAwesome + jQuery + axios for AJAX.

## Architecture (the big picture)
- **Routing:** `routes/web.php` is the main map; `api.php`, `channels.php`, `console.php` also exist. URLs don't always mirror view paths — confirm in `web.php`.
- **Controller layout is intentionally mixed:** grouped folders for feature areas (`KuBo/`, `Leave/`, `Overtime/`, `Reports/`, `Roles/`, `Profile/`) **plus** flat `*Ctrl.php` files, one per master-data screen (`companyCtrl.php`, `departmentCtrl.php`, `philhealthCtrl.php`, …). When touching a Settings screen, expect a flat `*Ctrl.php`; for payroll/leave/OT expect the larger grouped controllers.
- **Two Services locations:** `app/Services/` holds import + backup + schedule-request services (`AttendanceImportService`, `LeaveImportService`, `OvertimeImportService`, `DatabaseBackupService`, `ScheduleRequestService`, `DynamicMailManager`); `app/Http/Services/` holds domain services (`LeaveService`, `OvertimeService`, `RoleServices`). Business logic for these flows lives in the service, not the controller.
- **Email sending is provider-agnostic:** `App\Models\MailIntegrationSetting` stores admin-configured, encrypted credentials for whichever provider the client company uses (SMTP/Brevo, Mailgun, SES, Postmark — managed at Settings → Mail Integration). `App\Services\DynamicMailManager` reads the active row and configures a runtime `'dynamic'` Laravel mailer from it — any code that sends mail should go through `DynamicMailManager::mailer()`, not assume a fixed `.env` driver.
- **Maintenance mode** (Settings → Maintenance Mode, `maintenancemode` permission) is a custom app-level lockout, **not** Laravel's `php artisan down`. State lives in a single-row `App\Models\MaintenanceSetting` (`::current()` singleton, like `PayslipEmailSetting`): `is_active`, `scope` (`global` | `department`), `department_ids` (json, used when scope=department vs `emp_details.empDepID`), `message`, optional `starts_at`/`ends_at` window. `App\Http\Middleware\CheckMaintenanceMode` (alias `check.maintenance`, added to the main `['AuthCheck','check.employee.ip']` route group in `web.php`) blocks affected authenticated users with a 503 `errors/maintenance_active.blade.php` page (JSON 503 for ajax) — it does **not** log them out. Exemptions are permission-driven: `maintenancebypass` (assign to roles who keep working), `maintenancemode` (managers, so they can turn it off), the spatie `admin` role, and legacy super admin `users.role == 1`. `MaintenanceModeController@index/update` is the only backend; the screen is gated `can:maintenancemode`.
- **Department = company profile** (Settings → Departments, `departmentCtrl`, view `pages/management/departments.blade.php`, JS `public/js/settings/department.js`). This client runs a single company but uses the **Department** entity to represent it, so `departments` now carries a full company profile: `dep_name` (widened from `char(40)` to `varchar(191)`), `dep_address`, `dep_contact_phone`, `dep_email`, `dep_tin`, `dep_sss_employer_no`, `dep_philhealth_employer_no`, `dep_pagibig_employer_no`, `dep_logo_path`, `dep_description` (migration `..._add_profile_fields_to_departments_table`). Logos store to `public/img/departments/` (move-to-public_path convention, like company logos/COE signatures). **Multiple PDF documents** attach per department via `App\Models\DepartmentDocument` + `department_documents` table (FK `cascadeOnDelete`); files live under `public/docs/departments/<id>/`. `departmentCtrl@documents/uploadDocument/downloadDocument/deleteDocument` handle the Related-Documents panel (shown only when editing — needs an existing `depID`); upload is `mimes:pdf|max:10240`, and `getSize()` is read **before** `move()`. **File cleanup is the controller's job**, not the DB cascade: `deleteDocument` and `create_update`'s logo-replace `@unlink`, and `delete()` loops the department's docs + unlinks each file (and rmdirs the folder) before removing the row. `department` + `DepartmentDocument` both use `Auditable` → save via instance (`create()`/`forceFill()->save()`/instance `delete()`). **Payslip header shows the department name as the company:** both `pages/modules/payslip.blade.php` (on-screen) and `pages/modules/payslip_pdf.blade.php` (emailed TCPDF) render `optional($detail)->department->dep_name ?? ($company->comp_name ?? config('app.name'))` — department first, company/app-name fallback. Both render paths already eager-load `employee.empDetail.department` (PayrollController::payslip + SendPayslipEmailJob), so no N+1 — don't remove those `with()` clauses.
- **Payslip email automation** (`payslipemail` permission) lives mostly on the Payroll System screen, not its own page: the "Email Payslips" button in `pages/modules/payroll.blade.php` opens a modal (JS in `public/js/modules/payroll.js`) that calls `App\Http\Controllers\PayslipEmailController` (`/payroll/payslip/send`, `/payroll/payslip/status`, `/payroll/payslip/{payroll}/resend`, `/payroll/payslip-email/settings`). Sending itself is `App\Jobs\SendPayslipEmailJob` (queued, one per employee) → `App\Services\PayslipPdfService` (TCPDF, table-based `payslip_pdf.blade.php` template — NOT the grid/flexbox `payslip.blade.php` used for on-screen printing, TCPDF can't render that) for a password-protected PDF, password chosen by `App\Services\PayslipPasswordResolver` per `App\Models\PayslipEmailSetting` (birthdate from `emp_infos.empBdate`, or employee ID — there's no SSS-number column in this schema), then sent via `DynamicMailManager::mailer()`. Every attempt is logged to `App\Models\PayslipEmailLog`. Auto-send-on-approval is wired into `PayrollApprovalController@approve`.
- **Payroll domain** spans several models that must stay in sync: `Payroll`, `PayrollDetail`, `PayrollPeriod`, `PayrollLog` (computation breakdown shown on Payroll Logs screen), `PayrollApproval`. Government contributions are table-driven: `SssContribution`, `PhilhealthContribution`, `PagibigContribution`, `BirWithholdingTax` — editing a screen rarely means editing rates; the rates live in these tables/models.
- **Loan/charge deduction rule** (in `ContributionHelper::computeAll()` + `PayrollController::computePayroll()`): loans/charges (`Loan` rows: `salary`/`other`/`charges/penalty`/`cash_adv` company types + `sss`/`pagibig`/`philhealth` gov types) deduct on **EVERY cutoff**, not only end-of-month — each cutoff pays `min(monthly_amortization, balance)` (recurring charges deduct the full amount every cutoff) against the remaining balance until settled, gated by `status='active'`, balance/recurring, and `start_date <= pay_date`. **Trainees (TRN) ARE included** in loan deductions (money owed to the company) even though they're still excluded from gov't contributions/withholding tax. Each deduction writes a `LoanPayment` (for rollback on recompute/delete) and decrements the `Loan` balance — both gated only by `$canAffordLoans` now, no longer by `$isEndOfMonth`/`!== 'TRN'`. In contrast, **government contributions + withholding tax stay end-of-month-only and non-trainee-only** (computed inside `computeAll` under `if ($isEndOfMonth && !$isTrainee)`). Don't re-couple loans to the end-of-month or trainee gates.
- **Holiday-pay eligibility business rule** (in `PayrollController::computePayroll()`, REGULAR-holiday branch): if the employee didn't work the holiday and wasn't on leave/OB that day, look back to their **last SCHEDULED workday** before the holiday (via `EmployeeSchedule`, not just literal "yesterday" — skips rest days/weekends), capped at `$holidayLookbackDays` (14). Holiday pay is granted only if that prior scheduled day was Present, Approved **Paid** Leave (`LeaveDetail.leave_kind == '0'`), or OB; no scheduled day found within the cap ⇒ ineligible ⇒ holiday pay = 0. To keep this O(1) per employee (no N+1 across hundreds of employees × holidays), `$allLeaves`/`$allObs`/`$allSummaries` are bulk-fetched with their lower date bound widened to `$lookbackStart`, plus one extra bulk fetch `$allSchedulesForLookback` (deliberately separate from `$allSchedules`, which stays scoped to the cut-off and is iterated directly as "this period's attendance" — never widen that one). Per-employee, a sorted list of scheduled dates and an `$isPaidOnDate`/`$wasEligibleViaLastScheduledWorkday` closure pair are built once and reused for every holiday in the run. **Trainees are excluded from holiday pay entirely:** `$isTraineeForHoliday` (`$emp->empDetail->empClassification === 'TRN'`, the same code already used in `ContributionHelper`/`BirWithholdingTax` to skip gov't contributions and withholding tax — note trainees are **no longer** excluded from loan deductions, see the loan-deduction bullet below) is computed once per employee and gates the whole holiday-benefit grant — `$holidayPay` simply stays 0 for a `'TRN'` employee regardless of worked/leave/OB/lookback eligibility. **Holiday premium and OT are additive, not exclusive:** filing approved OT on the holiday does NOT cancel the holiday premium. The premium applies to the BASE worked/eligible day (regular +100% = `$dailyRate`, special +30% = `$dailyRate*0.3`); OT pays the EXTRA hours beyond schedule and is summed separately in `$totalOT` — they never overlap, so a non-trainee who works a holiday and files OT earns both. (Earlier code short-circuited on `$hasOtToday` and dropped the premium whenever any OT existed on the day — fixed.) `$hasOtToday` now only un-marks an otherwise-absent day; a single guarded `$absentDays--` covers both that case and the regular-holiday lookback case. No extra queries — `empClassification` is read off the already-eager-loaded `empDetail`.
- **Fixed bug (was: known pre-existing bug):** the daily loop's `$onLeave` lookup previously compared `$dateStr` against `$l->start_date`/`$l->end_date`, columns `LeaveDetail` doesn't have (only `date`), so it always evaluated false. Now compares `Carbon::parse($l->date)->format('Y-m-d') === $dateStr` directly, and distinguishes paid vs. unpaid leave via `leave_kind` (`'0'` = paid, `'1'` = unpaid — matches the leave-application form `<option value="0">Paid</option>` and every display/report/import; payroll previously had this **inverted**, so no paid leave was ever paid — fixed) — only paid leave counts toward `daysPresent` / holiday pay; unpaid leave is excused (not an absence) but not paid.
- **Login & forced password change:** sign-in (`loginCtrl@loginSystem`, AJAX `/loginSystem`) accepts **either an email OR a short `users.username`** — the field is detected with `filter_var(..., FILTER_VALIDATE_EMAIL)` and queried against the matching column (`max:50`, NOT `|email`, so long emails no longer fail validation). `username` is a nullable+unique column set on the Create/Edit Employee screens (`registerCtrl@create`/`@update`, `alpha_dash` + `Rule::unique('users','username')`); on **create**, a blank username auto-generates via `registerCtrl::generateUsername()`/`usernameCandidates()` (accent-folded, punctuation-stripped), and a one-time migration backfilled existing rows with the same rule. Candidate order: (1) `first-initial+surname` (`jdelacruz`); on collision (2) interchange to `surname-initial+firstname` (Realyn/Roselyn Cuenca → `crealyn`/`croselyn`); then (3) the employee ID; only then a numeric suffix. Edit keeps whatever is typed (blank ⇒ NULL). Existing accounts with NULL username still log in by email. **`users.must_change_password`** (boolean) gates `App\Http\Middleware\ForcePasswordChange` (alias `force.password`, in the main route group right after `AuthCheck`): a flagged user is redirected to the standalone `/force-password-change` screen (`profile/force_change.blade.php`) on every request — JSON 423 + `redirect` for ajax — until they set a new password via `ProfileController@forceUpdatePassword`, which clears the flag. A one-time migration set the flag for ALL existing users (new users default to false — not a recurring toggle). Password complexity (`Illuminate\Validation\Rules\Password::min(6)->mixedCase()->numbers()->symbols()`) lives in `ProfileController::passwordRules()` and is shared by both the forced flow and the self-service modal (`updatePassword`). **Admin password reset** ("forgot password") lives on the E-201 Personnel Viewer (`pages/management/e201.blade.php`): a "Reset Password" button (`public/js/modules/e201_admin.js`) POSTs to `admin/e201/reset-password/{user}` → `EmployeeRecordController@resetPassword`, which re-hashes the account to the same default temp password as new hires (`'123456'`) **and** sets `must_change_password = true` — so the employee is forced through the `force.password` flow to set their own password on next login; the admin never learns the final password.
- **Separated-employee login block** (data security): a separated employee (`emp_details.empStatus` not `'1'` — `'0'`=Resigned, `'2'`=End of Contract) **cannot access the system**. Enforced in two places, both **exempting** the legacy super admin (`users.role == 1`) and the spatie `admin` role (same exemption rule as Maintenance Mode, so an admin not marked Employed isn't locked out): (1) `loginCtrl@loginSystem` adds `emp_details.empStatus` to its select and, after the IP check, tears down the session (`Auth::logout()` + invalidate + regenerate) and returns `403` `'Your account is inactive. Please contact HR.'` for a non-exempt separated user; (2) `App\Http\Middleware\BlockSeparatedEmployee` (alias `block.separated`, added to the main authenticated group in `web.php` alongside `AuthCheck`/`check.maintenance`) catches anyone separated **while already logged in** — logs them out + redirects to `/auth/login` (JSON `403` + `redirect` for ajax), wrapped in try/catch so a DB hiccup fails open. This is why the COE for separated staff is HR-issued, not self-service.
- **"What's New" changelog** (no permission key — a standalone sidebar item in `layout/app.blade.php`, just after Home, which only renders when `public/changelog.json` exists). It is **git-driven, not hand-written**: `scripts/generate-changelog.php` (standalone PHP, no Laravel bootstrap — uses `proc_open` with an arg array so cmd.exe/sh quoting can't mangle the `%x1f`/`%x1e` git-log format) parses conventional-commit subjects into `public/changelog.json`. Because the deploy zip excludes `.git`, this **runs during CI** — currently **staging only** (`.github/workflows/staging.yml`, after composer install, before zip); `deploy.yml`/production is intentionally untouched, so the file (and thus the dropdown link) is absent in prod. `pageCtrl@whatsnew` reads/decodes the JSON; `pages/management/whatsnew.blade.php` groups entries by month and shows `feat`("New")/`fix`("Fixed") by default with a "Show technical changes" toggle for the rest. `public/changelog.json` is gitignored (build artifact). To enable on production later, add the same one-line generate step to `deploy.yml`.
- **Employee status / separation & flagging** (`manageemployeestatus` permission) also lives on the E-201 Personnel Viewer (`pages/management/e201.blade.php`), next to Reset Password: an "Update Status" button (`public/js/modules/e201_admin.js`) opens the `#updateStatusModal` that POSTs to `admin/e201/update-status/{user}` → `EmployeeRecordController@updateStatus`. **Two independent layers, one call:** (1) **Employment exit** reuses `emp_details.empStatus` (`'1'`=Employed, `'0'`=Resigned, `'2'`=End of Contract — payroll already filters `empStatus='1'`, so a non-`'1'` value auto-deactivates + excludes from payroll and flips the dossier badge to INACTIVE); a Resigned/EOC exit also writes `separation_reason`, `separation_date` (mirrored into legacy `empDateResigned`) and **freezes `years_rendered`** = `Carbon::parse(empDateHired)->floatDiffInYears(separation_date)` rounded to 2 dp. Re-selecting Employed reactivates and nulls those exit fields. (2) **Flag** (`flag_status` = `null` | `redflag` | `blacklist`, plus `flag_reason`/`flagged_at`/`flagged_by`) is **independent of employment state** — it can mark a still-active employee and never touches `empStatus`/payroll. All seven columns were added to `emp_details` (migration `..._add_separation_and_flag_to_emp_details.php`) and to `empDetail::$fillable`/`$casts`. `updateStatus` saves via the model **instance** (`$detail->save()`), NOT `where()->update()`, so the `Auditable` trait records the before→after diff. The full-form Edit Employee screen still has its own bare `emp_status` dropdown (no reason/years) — that path is unchanged. **(3) Offboarding clearance checklist** (same modal, shown only on exit): HR-attested checkboxes — **no file uploads** — for the requirements a separated employee must complete before a COE is issued. `App\Services\OffboardingClearanceService` is the single source of truth (`ITEMS` catalog + `applicableItems()`/`isComplete()`/`missingItems()`/`statusFor()`): `cl_resignation_letter` (Resigned only), `cl_office_notice` (EOC/Terminated only — both map to `empStatus='2'`), and `cl_clearance_form`/`cl_company_items`/`cl_quitclaim` (all separated). Columns (5 booleans + `clearance_refs` json for optional per-item notes + `cleared_by`/`cleared_at`) were added via `..._add_offboarding_clearance_to_emp_details.php`; `updateStatus` resets all flags then ticks only applicable+checked items (a non-applicable item can never be set), and **clears them on re-activation** alongside the exit fields. The JS shows only the items applicable to the chosen exit type; the dossier renders a read-only **Offboarding Clearance** card. This checklist is the gate read by the COE issuance flow (see the COE bullet).
- **Employment Documents — per-employee "201 file" on E-201** (`manageemployeedocuments` permission for upload/delete; **download/view needs only E-201 access**, like Reset Password). A third panel on the E-201 Personnel Viewer (`pages/management/e201.blade.php`, JS `public/js/modules/e201_admin.js`), alongside Reset Password + Update Status. One entity `App\Models\EmployeeDocument` (`employee_documents` table, migration `..._create_employee_documents_table`) — a **direct clone of the `DepartmentDocument` "Related Documents" pattern**, adapted to the employee. **Keyed on `user_id` (users.id) with `cascadeOnDelete`, NOT `empID`** — the E-201 admin action routes are `User`-model-bound and `empID` is a display-padded accessor (`User::empID` str_pads to 3 digits), so it's a poor FK/folder key; `empID` is stored denormalized for readability only. Files live under `public/docs/employees/<user_id>/` (move-to-public_path convention, same as department docs); accepts **PDF + images** (`mimes:pdf,jpg,jpeg,png|max:10240`, real extension preserved, size/mime/name captured **before** `move()`), plus a `doc_type` dropdown (Employment Contract / Government ID / Resume/CV / Certificate / Clearance / Other, whitelist `EmployeeRecordController::DOC_TYPES` kept in sync with the Blade `<select>`) + optional free-text label. `EmployeeRecordController@uploadDocument/downloadDocument/deleteDocument` (routes `admin/e201/documents/{user}` POST, `admin/e201/document/{document}/download` GET, `admin/e201/document/{document}` DELETE — upload/delete gated `can:manageemployeedocuments`). **No separate list endpoint:** `getEmployeeDetails` eager-loads `employmentDocuments` (User `hasMany`) and `renderDossier` paints them; after upload/delete the JS re-triggers the active row click to refresh, the same idiom as Update Status. **File cleanup is the controller's job** (`@unlink` before instance `delete()`); create/delete go through `Auditable` (instance save, like all E-201 writes). Permission seeded to admin/super/accessrights roles by migration `..._seed_manageemployeedocuments_permission` (summarylogs pattern) **and** is a `PagePermissionsEnum` case. Employees download via the same route (owner check passes).
- **Programs Management — tenure-milestone benefits** (`programs` permission, a **Workforce** menu item in `layout/app.blade.php`'s `$modulePages`, view `pages/modules/programs.blade.php`, JS `public/js/modules/programs.js`). Admin defines **milestone tiers by years of service** (`App\Models\TenureProgram`: `title`, `years_required`, `is_active`), each holding one or more **benefits** (`App\Models\TenureProgramBenefit`: `name`, `description` — e.g. "2 Years of Service" → "Bigas (Rice)"). Eligibility is **computed live, not stored**: `App\Services\TenureProgramService::eligibility()` does three bulk queries (active programs+benefits, active `empStatus='1'` employees with `empDateHired`, all grants) and cross-references in PHP (O(employees × programs), safe for the dashboard). **Tenure = `empDateHired->floatDiffInYears(today)`** (continuous, not whole-years); an employee "reaches" every tier whose `years_required <= tenure` (so at 4 yrs they qualify for both the 2-yr and 4-yr milestones). A `App\Models\TenureProgramGrant` row (unique per program+employee) marks a benefit as **granted**; no row = **pending**. `App\Http\Controllers\ProgramController` handles CRUD (`/programs/save` re-syncs benefits via delete-and-recreate in a transaction; saves the program via instance `forceFill()->save()` for `Auditable`), `eligibility()` (JSON for the screen), and `grant()`/`revoke()` (instance save/delete → `Auditable`). The screen shows stat cards, **Milestone Recipients** (filter All/Pending/Granted, Mark-Granted/Revert actions), **Upcoming Anniversaries** (next 60 days, `TenureProgramService::UPCOMING_DAYS`), and the milestone config table. **Wired to the HR Dashboard:** `HrDashboardController@index` injects the same `eligibility()` result (`$d['msStats']`/`$d['msPending']`/`$d['msUpcoming']`) into a "Tenure Milestones" card in `hr_dashboard.blade.php` — screen and dashboard never drift because both call the one service.
- **Notices / Memo module — disciplinary notices + suspension escalation** (`noticemanagement` permission for the HR side; the employee side is permission-less). One typed entity `App\Models\Notice` (`type` = `memo` informational | `disciplinary` violation; `category`, `title`, `body`, `issued_by`, `issued_at`, `status` = `active`|`void`, `is_read`/`read_at`). **Only active disciplinary notices count** toward escalation; memos and voided notices never do. `App\Services\NoticeService` holds the thresholds as constants — **`WARN=3`** (at-risk) and **`SUSPEND=4`** (over-limit): `evaluateEscalation($empId)` runs after any disciplinary notice is saved and, if the active-disciplinary count is `>= SUSPEND` **and** no pending recommendation exists, auto-creates one `App\Models\SuspensionRecommendation` (`status`=`pending`, `recommended_by`='System (auto)', auto-generated `reason`) — **idempotent** (won't stack multiple per employee). It never touches `empStatus`/`flag_status`; HR resolves recommendations manually (`dismissed`|`actioned`) via `NoticeController@resolveRecommendation`. `NoticeService::dashboard()` returns `over` (>=SUSPEND, shown as **names in an alert flash** on both the Notices admin page and the HR Dashboard top banner), `atRisk` (WARN..SUSPEND-1), `pendingRecs`, and tallies — same service feeds the admin screen and `HrDashboardController@index` (`$d['ntcOver']`/`$d['ntcAtRisk']`/`$d['ntcStats']` → flash banner + "Disciplinary Notices" card). **Admin** screen `pages/modules/notices.blade.php` (JS `public/js/modules/notices.js`): issue/edit/delete notices, employee picker (`/notices/employees`), filters, and the suspension-recommendations panel. **Employee** screen `pages/modules/my_notices.blade.php` (JS `public/js/modules/my_notices.js`, route `/pages/modules/mynotices`, controller `mine()`/`myList()`) — read-only list of the employee's own active notices; **opening it bulk-marks unread as read** (`is_read`/`read_at`, deliberately a bulk `update()` so it bypasses `Auditable` — read receipts aren't audited). **Badges:** an always-visible "My Notices" sidebar item (like What's New) and a topbar bell both show the employee's own unread count, computed once as `$myUnreadNotices` in `layout/app.blade.php` (mirrors the `$pendingLeaveCount` inline-count pattern; server-rendered, clears after the employee opens My Notices). `Notice` saves via instance (`forceFill()->save()` / `create()`) for `Auditable`.
- **Certificate of Employment (COE) module — request → HR e-sign approval → PDF download** (`coemanagement` permission for the HR side; the employee side is permission-less). One entity `App\Models\CoeRequest` (`employee_id`, `purpose`, `copies`, `date_needed`, `remarks`, `status` = `pending`|`approved`|`rejected`, `include_salary`, `certificate_no`, `snapshot` json, `signatory_name`/`signatory_title`/`signature_data`, `reviewed_by`/`reviewed_at`, `rejection_reason`). `App\Services\CoeService` is the single source of truth: **`requirements($empId)`** is the eligibility gate (active `empStatus='1'` + not blacklisted, complete profile = `empPos`/`empDepID`/`empDateHired` + name, required request fields, **no existing pending request**) returning `['ok','missing'[]]`; **`buildSnapshot($empId,$includeSalary)`** freezes the certified facts at approval time (name via `User::community_full_name`, position `pos_desc`, dept `dep_name`, company `comp_name`, classification `class_desc`, hire/regular dates, Regular-vs-Probationary, years of service, salary only if HR opts in) so the PDF never drifts if the record later changes; `nextCertificateNo()` issues `COE-YYYY-####`; `dashboard()` feeds both the screen and `HrDashboardController@index` (`$d['coeStats']`/`$d['coePending']` → "COE Requests" card). Employee gate is **re-enforced server-side** in `CoeController@store` (UI is not the guard). **HR approval** (`approve()`) stamps a **configured signatory's** e-signature — there is no draw-each-time pad. HR picks an active `App\Models\CoeSignatory` (validated `signatory_id`); its `name`/`title` and `signatureDataUri()` (the uploaded image read from `public/img/signatories/…` and base64-encoded) are **frozen** onto `signatory_name`/`signatory_title`/`signature_data`, so editing/deleting the signatory later never alters past certificates. The signature embeds as a data-URI `<img>` in the certificate. Signatories are maintained at **Settings → COE Signatories** (`App\Http\Controllers\CoeSignatoryController`, view `pages/management/coe_signatories.blade.php`, JS `coe_signatories.js`, route `pages/management/coe-signatories` gated `coemanagement`, reached via a "Manage Signatories" button on the COE screen): name + title + uploaded image (`image|mimes:jpeg,png,jpg|max:2048`, stored like profile photos via `move(public_path('img/signatories'))`) + `is_active`; the approve/issue pickers read `/coe/signatories/active`. **PDF** uses TCPDF via `App\Services\CoePdfService` (table/inline-style template `pages/modules/coe_pdf.blade.php`, same engine/constraints as `PayslipPdfService` — TCPDF can't render grid/flexbox; note an invalid PNG makes TCPDF fatal, but the canvas always emits valid PNG). **Download** route `/coe/{coe}/pdf` authorizes the owning employee OR a COE manager and only serves `approved` requests. **Admin** screen `pages/modules/coe.blade.php` (JS `coe.js`): stat cards, filterable requests table, approve modal (signature pad + include-salary toggle + signatory block), reject-with-reason. **Employee** screen `pages/modules/my_coe.blade.php` (JS `my_coe.js`, route `/pages/modules/mycoe`, controller `mine()`/`myList()`/`requirements()`/`store()`) — requirements checklist, request modal, own-request list with a Download button once approved. Always-visible "My COE" sidebar item (like My Notices); HR admin item is in `layout/app.blade.php`'s `$modulePages`. `CoeRequest` saves via instance (`forceFill()->save()` / `create()`) for `Auditable` (the large `signature_data` blob is in `auditIgnore`). **Separated employees (COE for ex-staff):** separated employees don't self-serve — they're **blocked from logging in** (see the login-block bullet), so HR issues their COE from the COE screen via `CoeController@issue` (button + modal on `coe.blade.php`; `separatedEmployees()` JSON feeds the picker with each one's clearance status). `issue()` is gated: the employee must be separated AND `OffboardingClearanceService::isComplete()` true (else `202` + the missing items), then it `create()`s an already-`approved` row (snapshot + `nextCertificateNo()` + signature) in one step — no pending phase. `buildSnapshot()` detects `empStatus !== '1'` → sets `is_separated`/`separation_date` and computes tenure/`employment_status` to the **separation date**, not today; `coe_pdf.blade.php` then phrases it in the **past tense** ("was employed … from {hired} to {separation}"). The issue modal uses the same signatory picker as approval (no pad).
- **Error Logs module — DB-backed exception capture + admin viewer** (`errorlogs` permission, a Settings/Management screen). Real server-side errors are recorded to the `error_logs` table so devs can browse and copy them (message, `file:line`, full stack trace, request URL/method, sanitized input, user, IP) instead of grepping `storage/logs/laravel.log`. Capture is hooked into the global `App\Exceptions\Handler::register()` `reportable()` callback: `Handler::shouldLogToDatabase()` filters out noise (404, method-not-allowed, access-denied, `TokenMismatchException`, `ValidationException`, `AuthenticationException`, any `HttpExceptionInterface` with status < 500 — and anything in `shouldntReport()`), everything else → `App\Models\ErrorLog::capture(Throwable)`. The callback does **not** call `->stop()`, so default file logging is unchanged. `ErrorLog::capture()` swallows all its own failures (logging must never break the request or recurse) and, unlike most models, **deliberately does NOT use `Auditable`** (no audit entry per captured error). Viewer `App\Http\Controllers\ErrorLogController@index` (filters: type/resolved/search/date) + `@resolve` (toggles the `resolved` flag via instance save); view `pages/management/errorlogs.blade.php` (mirrors `audit_trail.blade.php` — inline teal-token CSS, click-row modal with a consolidated `<pre>` copy block + Mark-Resolved). Mirrors the `audit_logs` feature end-to-end (migration seeds the `errorlogs` spatie permission for admin/super/accessrights roles). Sidebar item in `layout/app.blade.php`'s `$managementModules`, right after `auditlog`.
- **Summary Logs Management — back-door editor over computed attendance summaries** (`summarylogs` permission, a Workforce item in `layout/app.blade.php`'s `$modulePages`, view `pages/modules/summary_logs.blade.php`, JS `public/js/modules/summary_logs.js`, controller `SummaryLogsController`). Same filter/fetch shape as the Attendance Viewer report (`reportAttendanceCtrl@fetchAttendance`) but lists **only days that already have an `AttendanceSummary` row** — no partial-day synthesis — and adds an Edit modal over the computed figures: `total_hours` (Duration Gross), the manual-deduction total, `mins_late`, `mins_undertime`, `mins_night_diff`, `outpass_minutes` (Passout), `over_break_minutes`. **Net Duration is never stored** — always derived as gross − deductions/60 (the modal auto-syncs it; typing Net back-solves Gross; the server only accepts the stored fields). "Deductions (Min)" is really the sum of `AttendanceDeduction` rows (shared with the Adjustment Time screen): when the total is edited, the controller **consolidates** — deletes the summary's rows and writes one row with the new total (reason = admin note or `'Summary Logs manual adjustment'`, total 0 ⇒ no row). `AttendanceSummary` deliberately does **not** get the `Auditable` trait (every time-out rewrites a summary via `homeAttendance::updateDailySummary` — auditing those would flood the trail); instead `SummaryLogsController@update` records each manual edit explicitly with `AuditLog::record('updated', 'AttendanceSummary', id, diff)` including the deduction-total change and optional note. Permission is seeded to admin-ish roles by migration `2026_07_02_130000_seed_summarylogs_permission` (errorlogs pattern — no new table). **Payroll lock:** a day already covered by a generated `Payroll` row (`employee_id` + `payroll_start_date..payroll_end_date`, any status) is read-only — the payslip was computed from the old figures, so editing afterwards would silently desync attendance from pay. `update()` re-checks server-side and returns **423** with the covering period; `fetch()` bulk-loads overlapping payrolls (no N+1) and flags each row `payroll_locked`/`payroll_period` so the UI renders a lock badge instead of the Edit button. To fix a locked day: delete/regenerate that payroll run first, then edit, then re-compute.
- **Employee record** is split across `User`, `emp_info`, `empDetail`, `emp_family`, `emp_education`, `e201` — a single employee touches multiple models.
- **KuBo** is a self-contained social feed: `Community*` models, `KuBo/` controllers, `resources/views/kubo/`. The feed is JS infinite-scroll appending into `#kuboFeedContainer` (not server-rendered in full).

## How to navigate efficiently (for the AI)
1. Start from this file — don't re-scan the tree.
2. To find a screen's backend: view name in `pages/*` → matching `*Ctrl.php` or grouped controller → model(s) → route in `web.php` → service in `app/Services` or `app/Http/Services` if it's import/leave/OT/backup.
3. Only open the specific file you'll edit. Use `documentation.blade.php` for the full screen/permission inventory.
4. Keep this map current when structure changes.
